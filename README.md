# 给彦庭抓取每日政府公告功能

## 一、项目目标

本项目旨在为彦庭提供一个“每日政府公告抓取”能力，用于自动采集指定政府网站或公告栏目中的最新通知、公告、公示等信息，并按固定频率执行抓取、去重、汇总与通知，减少人工反复浏览多个网站的成本。

目标不是一次性做成复杂平台，而是先形成一个稳定、可扩展、可持续运行的日常抓取工具，为后续增加更多站点、筛选规则和消息推送能力打基础。

## 二、使用场景

### 1. 日常信息跟踪
彦庭希望每天自动获取政府网站上的新增公告，避免手动逐个网站检查。

### 2. 重点事项订阅
对于特定领域的公告，例如政策通知、项目申报、公示信息、采购招标或地方行政通知，希望能够持续跟踪。

### 3. 信息汇总与二次处理
抓取结果后续可能用于：
- 每日简报整理
- 飞书消息提醒
- 数据沉淀与检索
- 后续接入 AI 进行摘要、分类和优先级判断

## 三、核心功能

### 1. 公告源配置
支持配置一个或多个政府公告来源，包括站点首页、栏目页、列表页或搜索页。

### 2. 定时抓取
按照预设时间自动执行抓取任务，获取当天新增或最近更新的公告信息。

### 3. 结构化提取
从网页中提取并保存关键字段，建议至少包括：
- 标题
- 发布时间
- 来源单位
- 公告链接
- 公告正文摘要或正文内容
- 抓取时间

### 4. 新增公告识别
对抓取到的数据进行去重，识别哪些是首次出现的公告，哪些是历史已抓取内容。

### 5. 失败重试与异常记录
当目标网站短暂不可访问、页面结构变化或请求失败时，系统应保留失败记录并自动重试。

### 6. 结果输出
第一阶段建议先输出为便于查看和处理的形式，例如：
- 本地结构化文件
- 简单数据库记录
- 每日新增公告列表

### 7. 后续可扩展通知能力
为后续增加飞书通知、邮件提醒或日报汇总预留接口，但不要求在第一阶段全部完成。

## 四、抓取范围待确认项

以下范围需要在正式开发前进一步确认：

### 1. 目标站点范围
- 是单一政府网站，还是多个政府网站
- 是国家级、省级、市级还是区县级站点
- 是否包含事业单位、公共资源交易平台等扩展来源

### 2. 公告类型范围
- 普通通知公告
- 政策文件
- 招标采购
- 项目申报
- 人事公示
- 行政审批或执法公示

### 3. 时间范围与更新规则
- 仅抓取当日新增
- 抓取最近 N 天内容
- 是否关注“发布时间”与“更新时间”两种字段

### 4. 页面层级范围
- 只抓列表页标题和链接
- 进入详情页抓正文
- 是否需要附件链接识别与保存

### 5. 关键词筛选范围
- 是否抓全量公告
- 是否仅抓取包含特定关键词的公告
- 是否需要按主题分类

## 五、建议技术方案

以下为面向后续开发的建议方案草案：

### 1. 抓取执行方式
当前已切换为 **browser-agent 编排方案**（详见 `docs/browser-agent-architecture.md`）：
- 编排入口：`scripts/browser_agent_pipeline.py` 生成抓取计划
- 执行层：OpenClaw browser-agent subagent（MCP 真实浏览器），LLM 语义提取，无需维护 CSS 选择器
- 旧 Python Playwright 脚本保留作参考，不再作为主抓取方式

### 2. 页面采集策略
browser-agent 自动处理 JS 渲染和页面交互，统一用语义提取代替 CSS 选择器解析：
- 编排脚本生成任务模板 → browser-agent 逐站执行 → 结果合并归档
- 详见 `docs/browser-agent-architecture.md` 和 `docs/browser-agent-task-template.md`

### 3. 数据存储方式
MVP 阶段建议优先选择轻量存储方案，例如：
- SQLite
- JSON / CSV 文件

若后续站点增多、字段复杂、查询需求上升，再考虑升级为更完整的数据存储方案。

### 4. 任务调度方式
建议使用系统级定时调度或任务调度框架，保证可重复执行、可记录日志、可恢复。

### 5. 日志与监控
至少保留以下信息：
- 每次任务开始时间与结束时间
- 抓取站点
- 成功抓取数量
- 新增公告数量
- 失败页面与错误原因

### 6. 扩展性设计建议
建议在项目结构上区分：
- 站点配置
- 抓取逻辑
- 解析逻辑
- 存储逻辑
- 通知逻辑

这样便于后续新增站点时尽量少改动核心流程。

## 六、调度频率建议

可根据业务敏感度采用以下建议：

### 方案 A：每天固定 1 次
- 适合低频更新站点
- 成本低、实现简单
- 建议时间：工作日上午或中午

### 方案 B：每天固定 2～4 次
- 适合更新较频繁的公告栏目
- 可以更及时发现新增信息

### 方案 C：工作时段内按小时执行
- 适合对时效性要求较高的场景
- 需要更稳健的去重与容错机制

第一阶段建议从“每天 1～2 次”开始，先验证抓取稳定性，再视实际公告频率逐步调整。

## 七、去重与失败重试

## 1. 去重建议
建议至少组合以下维度进行去重判断：
- 公告链接
- 标题
- 发布时间

如目标站点存在同标题多次发布、同链接内容更新等情况，可进一步引入内容摘要或正文哈希作为辅助判断。

## 2. 重复数据处理策略
- 已存在的数据不重复写入
- 如发现内容更新，可记录“更新时间”并保留变更痕迹
- 对同一公告的重复抓取应保持幂等

## 3. 失败重试建议
- 单页面请求失败时，进行有限次数自动重试
- 单站点整体失败时，记录日志并在下一调度周期继续尝试
- 连续失败达到阈值后，应触发人工关注或告警

## 4. 异常场景考虑
- 网站超时
- 页面结构改版
- 反爬限制
- 编码异常
- 临时空页面或跳转异常

## 八、合规与风险提醒

### 1. 合法合规访问
抓取前应确认目标网站内容属于公开可访问范围，并遵守网站公开信息使用规则。

### 2. 尊重访问频率限制
避免高频请求，避免对目标站点造成明显负载压力。

### 3. robots 与站点策略
如果目标站点存在 robots、访问条款或明确限制，需要结合实际情况评估是否应抓取。

### 4. 数据用途边界
抓取数据应主要用于个人信息跟踪、内部整理与辅助分析，不应用于违规转载、商业滥用或侵犯权益的场景。

### 5. 反爬与封禁风险
部分政府网站可能存在访问限制、验证码或动态校验机制，技术方案需要避免激进策略，必要时降低频率或改为人工确认。

### 6. 内容准确性风险
公告页面可能存在延迟更新、字段缺失、链接变更等情况，抓取结果不能默认等同于官方最终解释，应保留原始链接供人工核对。

## 九、第一阶段 MVP 范围

第一阶段建议只做最小可用版本，目标是尽快跑通完整链路。

### MVP 建议范围
1. 先接入 1～2 个目标公告来源
2. 支持列表页抓取，并进入详情页获取正文或摘要
3. 保存基础字段：标题、时间、链接、来源、抓取时间
4. 完成基础去重逻辑
5. 支持定时执行
6. 支持失败日志与有限重试
7. 输出每日新增公告结果

### MVP 暂不包含
- 大规模多站点管理
- 复杂可视化后台
- 高级分类模型
- 自动摘要与智能打标
- 附件全文解析
- 复杂权限系统

## 十、待确认问题

1. 具体需要抓取哪些政府网站或栏目？
2. 是否有明确的关键词、主题或公告类型筛选要求？
3. 抓取结果最终希望输出到哪里：本地文件、数据库、飞书，还是其他渠道？
4. 对时效性的要求是什么：每日一次、每日多次，还是小时级？
5. 是否需要抓取公告正文、附件链接或附件内容？
6. 是否只关心新增公告，还是也关心历史公告补抓？
7. 是否需要保留公告变更历史？
8. 是否存在必须规避的网站、栏目或抓取限制？
9. 第一阶段优先接入的站点是哪些？
10. 后续是否要接入消息提醒、AI 摘要或分类能力？

## 十一、结论

该项目适合以“轻量、稳定、可扩展”的方式启动。第一阶段重点不是覆盖所有政府网站，而是先围绕明确的公告来源跑通“定时抓取—去重识别—结果输出—失败可恢复”这条主链路。待范围和优先级进一步确认后，再进入正式开发实施。
